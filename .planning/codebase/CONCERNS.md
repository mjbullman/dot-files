## Security Considerations

**Hardcoded absolute node binary path in `claude/settings.json`:**
- Risk: `claude/settings.json` contains 7 occurrences of the full absolute path `/Users/martinbullman/Library/Application Support/fnm/node-versions/v24.15.0/installation/bin/node`. This file is committed to a public repo. While the path itself is not a secret, it leaks username, machine layout, and the exact Node version used for GSD hooks. If the Node version is updated via fnm, all hook commands silently break.
- Files: `claude/settings.json`
- Current mitigation: None — paths are hardcoded strings
- Recommendations: Use `$(which node)` or `$(fnm which)` via a wrapper shell script instead of absolute paths, or use `env node` in a thin `.sh` launcher for each hook

**Local Ollama server IP address committed:**
- Risk: `opencode.jsonc` commits the LAN IP `http://192.168.1.180:11434/v1` pointing to a self-hosted Ollama server. This is a private network address, not a secret, but it creates a false assumption for any other machine cloning this repo: the model provider will silently fail without documentation that it requires a locally-reachable Ollama server at that address.
- Files: `opencode.jsonc`
- Current mitigation: None
- Recommendations: Document the Ollama requirement in README.md; or use a resolvable hostname (e.g. `ollama.local`) instead of a raw IP

## Hardcoded Machine-Specific Paths

**`neovim/lua/config/jdtls.lua` — JavaSE-22 path hardcodes the username:**
- Severity: MEDIUM
- Line 107: `path = '/Users/martinbullman/Library/Java/JavaVirtualMachines/openjdk-22.0.2/Contents/Home'`. The JavaSE-17 and JavaSE-21 entries use `/opt/homebrew/...` which is machine-agnostic. Only JavaSE-22 uses the user-specific path.
- Files: `neovim/lua/config/jdtls.lua:107`
- Impact: Neovim configuration is non-portable; the Java LSP runtime for SE-22 fails silently on any other machine or after a username change
- Fix approach: Install openjdk-22 via Homebrew (`brew install openjdk@22`) and use `/opt/homebrew/opt/openjdk@22/libexec/openjdk.jdk/Contents/Home`

**`neovim/lua/config/jdtls.lua` — jdtls cmd hardcodes Homebrew openjdk@21 path:**
- Severity: MEDIUM
- Line 60: `/opt/homebrew/opt/openjdk@21/libexec/openjdk.jdk/Contents/Home/bin/java` is a hardcoded absolute path. This is correct for Apple Silicon Homebrew (`/opt/homebrew`) but fails on Intel Macs (`/usr/local`) and Linux.
- Files: `neovim/lua/config/jdtls.lua:60`
- Fix approach: Use `vim.fn.exepath('java')` as a fallback, or derive via `io.popen('brew --prefix openjdk@21')` at startup

**`.zprofile` — Ruby gems path hardcodes version `4.0.0`:**
- Severity: LOW
- Line 39: `export PATH=/opt/homebrew/lib/ruby/gems/4.0.0/bin:$PATH`. When Ruby is upgraded via Homebrew, the gems directory version number changes and this path silently stops working.
- Files: `.zprofile:39`
- Fix approach: Use `$(ruby -e 'puts Gem.user_dir')/bin` instead of a hardcoded version path

**`.zprofile` — JetBrains Toolbox scripts path hardcodes username:**
- Severity: LOW
- Line 30: `export PATH="$PATH:/Users/martinbullman/Library/Application Support/JetBrains/Toolbox/scripts"`. Not portable to any other user account.
- Files: `.zprofile:30`
- Fix approach: Replace with `$HOME/Library/Application Support/JetBrains/Toolbox/scripts`

## Configuration Conflicts

**`vim.diagnostic.config()` called three times across separate files:**
- Severity: MEDIUM
- The last call wins in Neovim's load order. Three files each call `vim.diagnostic.config({...})` with different subsets of options:
  1. `neovim/lua/core/keymaps.lua:19` — sets `update_in_insert`, `float`
  2. `neovim/lua/config/lsp.lua:11` — sets `virtual_text`, `signs`, `underline`, `update_in_insert`, `severity_sort`, `float`
  3. `neovim/lua/config/inline-diagnostic.lua:24` — sets `virtual_text = false`
- The calls merge rather than replace, but this is fragile: options set in one file can be silently overridden by a later call, and the full effective diagnostic configuration is not visible anywhere.
- Fix approach: Consolidate all `vim.diagnostic.config({...})` into a single call in `neovim/lua/config/lsp.lua` and remove the partial calls from `keymaps.lua` and `inline-diagnostic.lua`

**`<C-j>` bound to two conflicting insert-mode actions:**
- Severity: MEDIUM
- In insert mode, `<C-j>` is bound to both:
  - `blink.cmp`: `select_next` (cycle down through completion items) — `neovim/lua/config/blink.lua:35`
  - `github/copilot.vim`: `<Plug>(copilot-accept-word)` — `neovim/lua/config/copilot.lua:44`
- Copilot is disabled by default (`vim.g.copilot_enabled = false`), so this only manifests when Copilot is manually enabled. At that point, `<C-j>` becomes ambiguous.
- Fix approach: Rebind one of the two. The Copilot accept-word binding is the logical candidate; change it to `<C-Right>` (currently used by Supermaven) or `<M-j>`.

**`<C-l>` bound to two conflicting insert-mode actions:**
- Severity: MEDIUM
- In insert mode, `<C-l>` is bound to both:
  - `supermaven`: `accept_suggestion` — `neovim/lua/config/supermaven.lua:8`
  - `github/copilot.vim`: `<Plug>(copilot-accept-line)` — `neovim/lua/config/copilot.lua:47`
- Both plugins run concurrently (Supermaven is always active; Copilot defaults to disabled but is togglable). When Copilot is enabled, `<C-l>` is ambiguous.
- Fix approach: Assign distinct bindings, e.g. Copilot accept-line → `<M-l>`.

**`<C-k>` bound to conflicting insert-mode actions:**
- Severity: LOW
- In insert mode, `<C-k>` is bound to both:
  - `blink.cmp`: `select_prev` — `neovim/lua/config/blink.lua:34`
  - LSP `signature_help` — `neovim/lua/config/lsp.lua:134`
- Blink's keymap will intercept `<C-k>` when the completion menu is open; the LSP binding wins otherwise. The `fallback` in blink means LSP should still work when blink is not active, but the interaction is implicit.
- Fix approach: This is a known blink.cmp pattern and is acceptable; document it or move LSP signature help to `<leader>k`.

## AI Completion Provider Overlap

**Three AI completion providers active simultaneously with overlapping responsibilities:**
- Severity: MEDIUM
- All three providers load on `VeryLazy` or eagerly:
  - **Codeium** (`neovim/lua/plugins/codeium.lua:8` `enabled = true`) — inline completion via Blink sources
  - **Supermaven** (`neovim/lua/plugins/supermaven.lua`) — inline ghost-text completion, always running
  - **GitHub Copilot** (`neovim/lua/plugins/copilot.lua`) — disabled by default but loaded
- Codeium and Supermaven both provide inline ghost-text at the same time. This doubles background network requests and binary processes, and creates UI conflicts where ghost-text from two providers may fight for display priority.
- Files: `neovim/lua/plugins/codeium.lua`, `neovim/lua/plugins/supermaven.lua`, `neovim/lua/plugins/copilot.lua`
- Fix approach: Pick one primary inline completion provider and set the others to `enabled = false`. Codeium is already integrated into Blink sources; Supermaven can be disabled via `neovim/lua/plugins/supermaven.lua` by adding `enabled = false`

## Performance Concerns

**`fastfetch` runs on every new terminal window at shell startup:**
- Severity: LOW
- `.zshrc:136`: `[[ $SHLVL -eq 1 ]] && fastfetch`. The `$SHLVL -eq 1` guard prevents fastfetch from running in nested shells (e.g. inside Neovim's `:term`), but it runs on every top-level terminal tab/window open. On a machine with many terminal tabs this is acceptable, but fastfetch still spawns system info queries on each new window.
- Files: `.zshrc:136`
- Impact: Minor startup latency (~50–150ms) per new terminal window

**Oh My Zsh + Starship + zsh-syntax-highlighting all load at shell startup:**
- Severity: LOW
- `.zshrc` loads the full Oh My Zsh framework (line 107), then immediately invokes Starship (line 139), even though Oh My Zsh's own theme system (`ZSH_THEME`) is disabled. Oh My Zsh adds measurable startup overhead (~100–300ms) even with only 3 plugins. The repo uses none of Oh My Zsh's unique features beyond plugin management.
- Files: `.zshrc:107`, `.zshrc:139`
- Fix approach: Consider replacing Oh My Zsh with a lightweight plugin manager (e.g. `zinit`, `sheldon`, or plain `fpath` sourcing) to reduce startup time. This is a significant refactor with low urgency.
